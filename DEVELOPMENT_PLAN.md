# 开发计划与自研范围

版本：v0.1  
日期：2026-05-27

## 1. 目标

本计划用于明确：

- 哪些能力可以优先复用开源项目。
- 哪些能力需要我们自己开发。
- 自研模块的难度、风险和开发顺序。
- 在多 agent 并行开发模式下，MVP 0 到 MVP 4 的落地路径。

本项目不应该重复实现成熟的文档解析、向量检索、RAG UI 或 OCR 底座。真正需要自研的是个人电脑文件系统上的“知识治理层”：全盘个人文件发现、隐私访问策略、资产地图、agent 工作流、安全清理建议和人类可浏览的知识结构。

## 2. 难度标注

- S：简单，主要是工程封装和规则配置。
- M：中等，需要稳定的数据结构、异常处理和跨平台适配。
- L：困难，需要较多策略设计、性能优化和真实数据验证。
- XL：很难，需要长期迭代，涉及安全、隐私、质量评估或复杂产品判断。

这里的难度不是人类工期。开发主要由 agent 完成时，难度更多代表：

- 需要多少真实环境验证。
- 是否容易误读敏感文件。
- 是否需要跨模块设计。
- 是否存在不可接受的误判风险。
- 是否需要用户确认真实偏好。

开发节奏按“开发波次”估算，而不是按周估算。一个开发波次可以包含多个并行 agent 同时推进不同模块，最后由主控 agent 合并、测试和校验。

## 4. 必须自研的核心模块

### 4.1 隐私访问策略层

难度：L

职责：

- 支持 `privacy.yaml`。
- 支持 `deny`、`metadata_only`、`no_embedding`、`allow`。
- 保证 `deny` 永远优先。
- 在读取文件内容前执行策略判断。
- 输出 dry-run 扫描计划。
- 记录访问审计日志。

为什么要自研：

- 开源 RAG/知识库一般假设用户主动导入文件。
- 本项目是全盘或半全盘扫描，必须先解决“什么不能读”。
- 这是产品信任的基础。

主要风险：

- 规则优先级错误导致误读敏感文件。
- Windows 路径、软链接、快捷方式、云盘路径处理复杂。
- 应用数据目录容易包含隐私数据库。

### 4.2 Windows-first 文件扫描器

难度：L

职责：

- 枚举磁盘和用户目录。
- 识别个人文件候选。
- 排除系统目录、缓存、安装包、构建产物等噪声。
- 支持增量扫描。
- 支持文件移动、删除、修改检测。
- 生成 `inventory.sqlite`。

为什么要自研：

- 通用文件搜索工具不会理解本项目的“个人知识价值”。
- GNO 等项目不适合直接指向整个系统根目录。
- 需要和隐私策略、生命周期建议、资产地图联动。

主要风险：

- 全盘扫描噪声巨大。
- 文件数量多时性能和 IO 压力大。
- 权限错误、锁定文件、云盘占位文件都要处理。

### 4.3 默认排除规则库

难度：M

职责：

- 维护 Windows/macOS/Linux 的系统噪声路径。
- 维护开发项目噪声路径，例如 `node_modules`、`.venv`、`dist`、`build`。
- 维护危险文件类型，例如 `.pem`、`.key`、`.env`。
- 维护安装包、缓存、临时文件识别规则。

为什么要自研：

- 这是项目体验好坏的关键。
- 规则需要面向个人电脑真实环境长期迭代。

主要风险：

- 过度排除会漏掉有价值文件。
- 排除不足会造成扫描结果不可用。

### 4.4 Inventory 数据库

难度：M

职责：

- 存储文件元数据、隐私策略命中、解析状态、生命周期状态。
- 支持增量扫描。
- 支持文件 ID、hash、路径变化追踪。
- 支持后续和 GNO/OpenKB/本地索引同步。

为什么要自研：

- 需要成为整个系统的事实表。
- 开源项目的内部数据库不一定适合做跨模块协调。

主要风险：

- 路径变化和重复文件处理容易混乱。
- hash 策略影响性能。

### 4.5 解析编排层

难度：M

职责：

- 根据文件类型选择 Direct Read、MarkItDown、Docling 或自定义 parser。
- 记录 parser 版本、错误、耗时、输出路径。
- 避免重复解析未修改文件。
- 限制大文件、坏文件、压缩包递归等风险。

为什么要自研：

- MarkItDown 和 Docling 是解析器，不是完整调度系统。
- 需要把解析结果接到 inventory、summary、index、wiki。

主要风险：

- 解析失败类型多。
- 大 PDF、坏 Office、编码异常、压缩包炸弹都要兜底。

### 4.6 本地摘要、标签和分类管线

难度：L

职责：

- 为文件生成短摘要、长摘要、主题、实体、标签。
- 生成项目、人物、时间、文件类型、动作状态等分类。
- 支持本地 LLM 或规则优先的降级方案。
- 把结果写入 summaries、tags、taxonomy。

为什么要自研：

- 这是“知识库能被人理解”的核心。
- 开源 RAG 多数只解决问答，不解决长期资产盘点。

主要风险：

- 本地模型质量不稳定。
- 标签容易发散，需要标准化和合并。
- 文件多时成本和时间压力大。

### 4.7 人类可浏览资产地图

难度：L

职责：

- 生成标签树、主题树、项目树、人物页、时间线。
- 支持逐级点开查看文件数量、摘要、代表文件。
- 支持未分类、高价值、长期未用、归档候选、删除候选视图。
- 第一版可以输出静态 Markdown/HTML，后续做 Web UI。

为什么要自研：

- 这是本项目区别于普通 RAG 的关键体验。
- 用户不应该只能通过搜索框知道自己有什么。

主要风险：

- 分类层级设计不好会变成另一套混乱目录。
- 自动标签需要人工可修正。
- 静态输出和后续 UI 之间要保持数据结构一致。

### 4.8 Agent Skill / 工作流编排

难度：M

职责：

- 编写 OpenClaw / Hermes / Codex 可用的 skill。
- 定义 agent 什么时候搜索、什么时候读摘要、什么时候查看原文件。
- 封装 `pfkb status/search/ask/summarize/review-deletes/archive-plan` 等命令。
- 对接 GNO MCP 或 CLI。

为什么要自研：

- GNO 提供通用 skill，但本项目需要“个人文件治理”语义。
- Hermes / OpenClaw 的使用习惯可能需要单独适配。

主要风险：

- agent 误用工具导致读太多文件。
- skill 描述太宽会频繁误触发，太窄又不好用。

### 4.9 清理、归档和删除建议

难度：XL

职责：

- 识别疑似重复文件。
- 识别安装包、缓存、低价值文件。
- 识别长期未用但可能有价值的文件。
- 生成归档建议和删除候选报告。
- 删除前只生成计划，不自动删除。
- 后续支持隔离区、回收站、manifest。

为什么要自研：

- 文件删除是高风险动作，开源 RAG 项目通常不涉及。
- 这里需要结合文件内容、路径、时间、重复度、用户习惯综合判断。

主要风险：

- 误删风险极高。
- “低价值”判断很主观。
- 需要非常清楚的可解释性和回滚机制。

### 4.10 应用个人数据适配器

难度：XL

职责：

- 识别哪些 app 数据可以安全读取。
- 支持浏览器书签、导出聊天记录、邮件、笔记应用、IDE 工作区等。
- 优先读取用户导出的数据，不直接读取高风险内部数据库。
- 每个 app adapter 都要有单独隐私策略。

为什么要自研：

- 各应用数据格式差异很大。
- 个人数据隐私风险高。
- 这部分没有统一开源底座可直接解决。

主要风险：

- 法律、隐私、数据损坏风险。
- 应用版本变化导致解析失效。
- 直接读取数据库可能破坏用户信任。

### 4.11 质量评估和审计

难度：L

职责：

- 评估扫描命中率、排除规则命中率、解析成功率。
- 评估标签质量、摘要质量、检索命中率。
- 输出访问日志、解析日志、错误报告。
- 支持用户反馈修正标签和规则。

为什么要自研：

- 本项目长期运行，必须知道它做得好不好。
- 没有审计能力，隐私和删除建议都不可信。

主要风险：

- 评估指标设计困难。
- 用户反馈闭环需要产品化。

## 5. Agent 并行开发模式

### 5.1 角色分工

项目可以按多 agent 并行推进：

| Agent 角色 | 负责内容 | 可并行程度 |
| --- | --- | --- |
| 主控 agent | 架构决策、任务拆分、合并代码、最终验证 | 串行核心 |
| 扫描 agent | 文件扫描、路径规范化、inventory 写入 | 高 |
| 隐私 agent | `privacy.yaml`、策略优先级、dry-run、审计日志 | 高 |
| 规则 agent | 默认 excludes、危险扩展名、系统目录规则 | 高 |
| 解析 agent | MarkItDown / Docling 编排、解析状态 | 中 |
| 检索 agent | GNO 接入、CLI/MCP/skill 适配 | 中 |
| 分类 agent | 摘要、标签、taxonomy、资产地图 | 中 |
| QA agent | 测试、真实目录样例、误判分析、报告 | 高 |
| 文档 agent | README、使用说明、配置说明、开发记录 | 高 |

### 5.2 串行瓶颈

以下任务不能完全并行：

- 隐私策略必须先于任何内容读取。
- Inventory schema 需要先稳定到足够支撑扫描、解析和索引。
- 清理/删除建议必须等扫描、摘要、重复检测、隐私策略都稳定后再做。
- app 个人数据适配器必须逐个确认隐私边界。
- 真实文件系统验证必须在本机环境中迭代，不能只靠模拟数据。

### 5.3 Agent 执行单位

本文后续使用“开发波次”估算：

- 0.5 个波次：小范围文档、配置、规则或单模块补丁。
- 1 个波次：一个可运行模块或一个端到端薄切片。
- 2 到 3 个波次：多个模块联动，并需要测试和重构。
- 4 个波次以上：涉及真实数据反馈、安全边界或质量评估的复杂能力。

## 6. 开发阶段计划

### MVP 0：扫描、隐私和 Inventory

目标：

- 能安全地扫描少量真实个人目录。
- 用户能清楚看到哪些文件会被读取、哪些不会。
- 建立后续所有模块依赖的 inventory。

交付：

- `privacy.yaml`
- `excludes.yaml`
- Windows-first scanner
- dry-run `scan-plan.md`
- `inventory.sqlite`
- `access-log.jsonl`
- 扫描统计报告

自研模块：

- 隐私访问策略层：L
- Windows-first 文件扫描器：L
- 默认排除规则库：M
- Inventory 数据库：M

Agent 执行估算：

- 1 到 2 个开发波次可做出可运行原型。
- 其中扫描、隐私策略、默认规则、inventory 可以由多个 agent 并行实现。
- 真实目录验证和规则修正需要额外 0.5 到 1 个波次。

### MVP 1：解析和本地检索

目标：

- 常见文件能被解析成文本/Markdown。
- Agent 可以搜索并读取本地知识库上下文。

交付：

- Direct text parser
- MarkItDown 解析接入
- Docling 可选接入
- 解析状态表
- GNO collection / MCP / skill 接入
- `pfkb search`
- `pfkb explain-file`

自研模块：

- 解析编排层：M
- GNO 接入适配：M
- Agent Skill / 工作流编排：M

Agent 执行估算：

- 1 个开发波次可完成端到端薄切片。
- 解析 agent 和检索 agent 可以并行，但需要主控 agent 统一输出目录、状态表和 CLI。

### MVP 2：摘要、标签和资产地图雏形

目标：

- 文件不用打开也能知道是什么。
- 用户可以按标签、主题、项目、文件类型逐级浏览。

交付：

- per-file summary
- tags/entity/topic extraction
- taxonomy 数据结构
- 静态 Markdown/HTML 资产地图
- `pfkb tags`
- `pfkb summarize`

自研模块：

- 本地摘要、标签和分类管线：L
- 人类可浏览资产地图：L
- 质量评估和审计初版：M

Agent 执行估算：

- 2 到 3 个开发波次。
- 摘要、标签、taxonomy、静态资产地图可以并行做，但最后需要一次统一的数据模型收敛。

### MVP 3：清理和归档助手

目标：

- 帮用户安全发现可归档、疑似重复、低价值、安装包/缓存文件。
- 只生成建议和复核报告，不自动删除。

交付：

- duplicate candidate report
- archive candidates
- delete candidates
- review report
- manifest 草案
- `pfkb review-deletes`
- `pfkb archive-plan`

自研模块：

- 清理、归档和删除建议：XL
- 质量评估和审计增强：L

Agent 执行估算：

- 2 到 4 个开发波次。
- 代码实现可并行，但误判分析、解释性、manifest 和回滚设计是主要瓶颈。
- 第一版只输出建议报告，不进入自动删除。

### MVP 4：Wiki 编译和长期维护

目标：

- 把高价值文件沉淀成长期可读的知识体系。
- Agent 和人类都可以围绕 wiki 使用知识库。

交付：

- wiki 目录结构
- concepts / projects / people / timeline pages
- OpenKB 思路接入或自研轻量 wiki generator
- watch / incremental update
- 本地 Web UI 或更完善的静态站点

自研模块：

- 人类可浏览资产地图增强：L
- Wiki 编译适配：L
- Agent 工作流增强：M

Agent 执行估算：

- 2 到 4 个开发波次。
- Wiki 生成、页面结构、导航、agent 可读摘要可以并行推进。
- 适合在 MVP 0-3 数据稳定后集中打磨。

### MVP 5：应用个人数据适配器

目标：

- 在隐私可控前提下，把 app 内有价值的个人数据纳入知识库。

交付：

- 浏览器书签 adapter
- 聊天导出 adapter
- 邮件导出 adapter
- 笔记应用导出 adapter
- 每个 adapter 的独立隐私说明和 dry-run 报告

自研模块：

- 应用个人数据适配器：XL
- 隐私访问策略增强：L

Agent 执行估算：

- 每个 app adapter 可以独立拆成 0.5 到 2 个开发波次。
- 不建议放进第一版核心路径，因为隐私边界和真实格式差异是主要瓶颈。

## 7. 推荐优先级

第一优先级：

1. 隐私访问策略层。

3. Inventory 数据库。

1. 解析编排层。

第三优先级：

1. 摘要和标签。
2. 人类可浏览资产地图。

1. 归档和删除建议。
2. Wiki 编译。

## 8. 最小可开工任务拆分

### Task 1：项目骨架

难度：S

- 建立 `src/`、`configs/`、`data/`、`docs/`。
- 定义 CLI 命令结构。
- 确定 Python 或 Node 技术栈。

### Task 2：隐私配置和策略判断

难度：L

- 读取 `privacy.yaml`。
- 实现 deny / metadata_only / no_embedding / allow。
- 对路径、扩展名、glob 规则做匹配。
- 输出策略命中原因。

### Task 3：Dry-run 扫描

难度：M

- 只遍历，不读取文件内容。
- 输出 `scan-plan.md`。
- 统计允许读取、仅元数据、完全跳过的数量。

### Task 4：Inventory SQLite

难度：M

- 设计 schema。
- 写入扫描结果。
- 支持重复运行更新。

### Task 5：真实目录验证

难度：M

- 用 Desktop/Documents/Downloads 小范围测试。
- 调整默认排除规则。
- 记录误排除和漏排除。

完成这 5 个任务后，项目就有了第一块可信地基。

## 9. 当前判断

开源项目已经解决了很多“知识库基础设施”问题，但没有解决个人电脑全盘文件的产品治理问题。

因此，自研重点应该放在：

- 让用户信任它不会乱读文件。
- 让扫描结果从一开始就干净。
- 让知识结构同时给人和 agent 使用。
- 让清理建议可解释、可复核、可回滚。

只要 MVP 0 做扎实，后面的解析、检索、wiki、agent 接入都可以站在开源项目上快速推进。
